## 1. 基础用法

支持两种占位符包裹形式：
- `{placeholder_name}`
- `%placeholder_name%`

示例：
- 输入：`你好, {player_name}!`
- 输出：`你好, Steve!`

---

## 2. 参数分流规则

### 2.1 通用语法

```text
{token:参数部分}
```

推荐标准写法（强烈建议）：

```text
{token:业务参数...|格式化参数...}
```

### 2.2 分隔符含义

| 分隔符 | 含义 | 示例 |
|---|---|---|
| `:` | 分隔 token 与参数 | `{player_ping:...}` |
| `,` | 分隔同一层参数项 | `precision=1,map=...` |
| `;` | 分隔 `map` / `bool_map` / `char_map` / `regex_map` 内规则 | `map=>200:高;>100:中;低` |
| `|` | 左侧业务参数，右侧格式化参数 | `{total_entities:exclude_drops|precision=0}` |

### 2.3 关键规则（与实现一致）

1. 使用 `|` 时：
- 左侧全部视为业务参数（传给占位符 `evaluateWithArgs`）。
- 右侧全部视为格式化参数（`precision/map/...`）。

2. 不使用 `|` 时：
- `precision=`, `map=`, `color_format=`, `bool_map=`, `char_map=`, `regex_map=`, `json_map=`, `eq_eps=` 会被识别为格式化参数。
- 其他参数会被当作业务参数。

3. 这意味着：
- 想传业务参数时，建议始终写 `|`，避免误判。
- 想写颜色阈值（如 `50,§a,100,§e,§c`）时，也建议放在 `|` 右侧，否则可能被当成业务参数。

### 2.4 转义与复杂参数

参数分割器支持：
- 引号包裹：`'...'` / `"..."` 内部的 `,`、`;` 不会分割。
- 转义字符：`\` 可转义分隔符，例如 `\,`、`\;`。
- 括号和花括号层级：`(...)`、`{...}` 内部的分隔符会按层级处理，适合 `json_map`。

## 3. 格式化参数

以下参数均写在格式化区（推荐放在 `|` 右侧）。

### 3.1 `precision`

语法：
```text
precision=<小数位数>
```

示例：
- 输入：`{player_pos_x:|precision=2}`
- 输出：`123.46`

说明：
- 仅当当前值为纯数值字符串时生效。

### 3.2 `map`

语法：
```text
map=<规则1>;<规则2>;...;<默认输出>
```

规则格式：
```text
<操作符><阈值>:<输出文本>
```

支持操作符：`>` `<` `=` `>=` `<=` `!=`

可用占位符：
- `{value}`：原始求值结果（数值字符串）

示例：
- 输入：`{player_ping:|map=>200:§c高延迟({value});>100:§e一般({value});§a正常({value})}`
- 输出（175）：`§e一般(175)`

重要行为：
- 若匹配输出中不写 `{value}`，系统会自动把原值拼在后面（即“前缀+原值”）。
- `map` 启用时，颜色阈值规则不会再执行。

### 3.3 `eq_eps`

语法：
```text
eq_eps=<浮点比较容差>
```

用途：
- 调整 `map` 中 `=` / `!=` 的浮点比较容差。

### 3.4 `bool_map`

语法：
```text
bool_map=<原值1>:<目标值1>;<原值2>:<目标值2>
```

示例：
- 输入：`{actor_is_alive:|bool_map=true:§a存活;false:§c死亡}`

### 3.5 `char_map`

语法：
```text
char_map=<原串1>:<目标串1>;<原串2>:<目标串2>
```

示例：
- 输入：`{player_name:|char_map=-:_}`

### 3.6 `regex_map`

语法：
```text
regex_map=<正则1>:<替换1>;<正则2>:<替换2>
```

说明：
- 替换串支持 `$1`、`$2` 等捕获组。

示例：
- 输入：`{player_name:|regex_map=^Player_(\w+)_(\d+)$:User-$1-$2}`

### 3.7 `json_map`

语法：
```text
json_map=<JSON对象>
```

示例：
- 输入：`{item_name:|json_map={"diamond:pickaxe":"§b钻石镐","apple":"§c苹果"}}`

### 3.8 颜色规则

1) 单色：
```text
{player_name:|§e}
```

2) 阈值颜色：
```text
{player_ping:|50,§a,100,§e,§c}
```

逻辑：
- 依次判断 `<阈值`，命中即使用对应颜色。
- 都不命中则使用最后的默认颜色。

3) 自定义输出格式：
```text
color_format=<模板>
```

模板占位符：
- `{color}`：选中的颜色码
- `{value}`：当前值

示例：
- 输入：`{player_ping:|50,§a,100,§e,§c,color_format=§7[§r{color}{value}§7]}`

## 4. 执行顺序与组合规则

单个占位符的处理顺序：

1. 用业务参数执行占位符求值。
2. `map`
3. `precision`
4. `bool_map`
5. `char_map`
6. `regex_map`
7. `json_map`
8. 颜色规则（仅在未启用 `map` 时执行）

组合建议：
- 需要“条件文本”时优先用 `map`。
- 需要“纯颜色分级”时不要启用 `map`，直接用颜色阈值。
- 业务参数和格式化参数同时存在时，优先使用 `业务|格式化` 的明确写法。

## 5. 实战模板

### 5.1 纯颜色分级 Ping

```text
{player_ping:|50,§a,100,§e,§c}ms
```

### 5.2 带文本状态的 Ping

```text
{player_ping:|map=>200:§c高延迟({value});>100:§e一般({value});§a正常({value})}ms
```

### 5.3 布尔值本地化

```text
{actor_is_alive:|bool_map=true:是;false:否}
```

### 5.4 复杂键映射（含冒号）

```text
{item_name:|json_map={"diamond:pickaxe":"钻石镐","minecraft:apple":"苹果"}}
```

## 6. 常见错误

### 6.1 token 与参数之间写成逗号

错误：
```text
{player_ping,map=>200:§c;>100:§e;§a}
```

正确：
```text
{player_ping:|map=>200:§c;>100:§e;§a}
```

### 6.2 `map` 规则之间误用逗号

错误：
```text
{player_ping:|map=>200:§c,>100:§e,§a}
```

正确：
```text
{player_ping:|map=>200:§c;>100:§e;§a}
```

### 6.3 不写 `|` 导致参数分流不符合预期

现象：
- 本想写格式化参数，结果被当成业务参数传给占位符。

建议：
- 业务参数与格式化参数并存时，固定写成 `{token:业务|格式化}`。
- 只有格式化参数时，也建议写成 `{token:|格式化}`。
